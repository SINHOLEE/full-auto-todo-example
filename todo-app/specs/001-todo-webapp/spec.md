## User Scenarios & Testing *(mandatory)*

### User Story 1 - 할 일 추가 및 조회 (Priority: P1)

사용자는 웹 브라우저에서 할 일 앱에 접속하여 할 일을 입력하고 추가할 수 있다. 앱을 새로 열거나 새로고침해도 이전에 추가한 할 일 목록이 그대로 유지된다.

**Why this priority**: 할 일 추가 및 조회는 Todo 앱의 가장 핵심 기능으로, 이것만 있어도 앱으로서 가치를 전달할 수 있다.

**Independent Test**: 입력창에 할 일을 입력하고 추가 버튼을 클릭한 후, 목록에 항목이 나타나고 새로고침 후에도 유지되는지 확인으로 독립 테스트 가능.

**Acceptance Scenarios**:

1. **Given** 앱이 열려 있을 때, **When** 사용자가 입력창에 텍스트를 입력하고 추가 버튼을 클릭하면, **Then** 할 일 목록에 새 항목이 추가된다.
2. **Given** 할 일이 추가된 상태에서, **When** 페이지를 새로고침하면, **Then** 이전에 추가한 할 일 목록이 그대로 표시된다.
3. **Given** 앱이 열려 있을 때, **When** 빈 입력창에서 추가 버튼을 클릭하면, **Then** 빈 항목은 추가되지 않는다.

---

### User Story 2 - 할 일 완료 처리 (Priority: P2)

사용자는 할 일 항목의 체크박스를 클릭하여 완료 상태를 토글할 수 있다. 완료된 항목은 시각적으로 구분되어 표시된다.

**Why this priority**: 완료 처리는 Todo 앱의 핵심 인터랙션이며 사용자가 진행 상황을 추적하는 데 필수적이다.

**Independent Test**: 체크박스를 클릭했을 때 완료 상태가 변경되고 새로고침 후에도 상태가 유지되는지 확인으로 독립 테스트 가능.

**Acceptance Scenarios**:

1. **Given** 미완료 할 일이 목록에 있을 때, **When** 사용자가 체크박스를 클릭하면, **Then** 해당 항목이 완료 상태로 변경되고 시각적으로 구분된다.
2. **Given** 완료된 할 일이 있을 때, **When** 사용자가 체크박스를 다시 클릭하면, **Then** 해당 항목이 미완료 상태로 되돌아간다.
3. **Given** 완료 상태가 변경된 후, **When** 페이지를 새로고침하면, **Then** 변경된 완료 상태가 유지된다.

---

### User Story 3 - 할 일 삭제 (Priority: P3)

사용자는 각 할 일 항목에 있는 삭제 버튼을 클릭하여 원하지 않는 항목을 제거할 수 있다.

**Why this priority**: 삭제 기능은 목록 관리에 중요하나 추가/완료 기능이 먼저 동작해야 의미가 있다.

**Independent Test**: 삭제 버튼 클릭 시 항목이 목록에서 사라지고 새로고침 후에도 삭제 상태가 유지되는지 확인으로 독립 테스트 가능.

**Acceptance Scenarios**:

1. **Given** 할 일 목록에 항목이 있을 때, **When** 사용자가 특정 항목의 삭제 버튼을 클릭하면, **Then** 해당 항목이 목록에서 즉시 사라진다.
2. **Given** 항목이 삭제된 후, **When** 페이지를 새로고침하면, **Then** 삭제된 항목이 다시 나타나지 않는다.

---

### User Story 4 - 인터넷을 통한 앱 접근 (Priority: P4)

사용자는 인터넷에 배포된 URL로 어디서든 Todo 앱에 접근하여 사용할 수 있다.

**Why this priority**: 배포는 앱 완성의 마지막 단계로, 로컬 기능이 모두 동작한 후 진행한다.

**Independent Test**: 배포된 URL로 접속하여 할 일 추가/완료/삭제가 정상 동작하는지 확인으로 독립 테스트 가능.

**Acceptance Scenarios**:

1. **Given** 앱이 배포된 상태에서, **When** 사용자가 제공된 URL로 접속하면, **Then** 앱이 정상적으로 로드된다.
2. **Given** 배포된 앱에서, **When** 사용자가 할 일을 추가/완료/삭제하면, **Then** 변경사항이 데이터베이스에 반영되고 유지된다.

---

### Edge Cases

- 할 일 제목이 매우 긴 경우(100자 이상) UI가 깨지지 않고 처리되어야 한다.
- 네트워크 오류 시 사용자에게 적절한 피드백이 표시되어야 한다.
- 빈 문자열 또는 공백만 있는 입력으로 추가를 시도하면 차단되어야 한다.

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: 시스템은 사용자가 텍스트를 입력하고 할 일을 추가할 수 있어야 한다.
- **FR-002**: 시스템은 앱 로드 시 데이터베이스에서 기존 할 일 목록을 불러와야 한다.
- **FR-003**: 시스템은 각 할 일 항목의 완료 상태를 토글할 수 있어야 한다.
- **FR-004**: 시스템은 사용자가 특정 할 일 항목을 삭제할 수 있어야 한다.
- **FR-005**: 시스템은 모든 데이터 변경사항(추가/수정/삭제)을 영구 저장소에 즉시 반영해야 한다.
- **FR-006**: 시스템은 빈 텍스트 또는 공백으로 할 일을 추가하는 것을 방지해야 한다.
- **FR-007**: 앱 화면에는 "나의 할 일" 제목, 텍스트 입력창, 추가 버튼, 할 일 목록(항목별 체크박스 및 삭제 버튼)이 포함되어야 한다.

### Key Entities

- **Todo 항목**: 사용자의 할 일을 나타내며 고유 ID, 제목(텍스트), 완료 여부(참/거짓), 생성 시각을 속성으로 가진다.
- **할 일 목록**: Todo 항목들의 집합으로, 생성 시각 기준 내림차순(최신 항목이 상단)으로 표시된다.

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 사용자가 30초 이내에 할 일을 추가하고 목록에서 확인할 수 있다.
- **SC-002**: 페이지 새로고침 후 3초 이내에 기존 할 일 목록이 표시된다.
- **SC-003**: 추가(Create), 조회(Read), 완료 상태 변경(Update), 삭제(Delete) 4가지 기능이 모두 데이터베이스와 연동되어 정상 동작한다.
- **SC-004**: 배포된 URL로 인터넷 어디서든 앱에 접속하여 전체 기능을 사용할 수 있다.
- **SC-005**: 데이터가 영구 저장되어 새로고침 또는 재접속 후에도 유지된다.

## Assumptions

- 사용자는 계정 로그인 없이 공유된 단일 할 일 목록을 사용한다 (인증 기능은 범위 외).
- 학습 목적으로 데이터베이스 보안 정책(RLS)은 비활성화 상태로 진행한다.
- 할 일 텍스트 수정(Edit) 기능은 v1 범위 외이며, 완료 상태 토글만 지원한다.
- 모바일 반응형 UI는 선택사항이며, 데스크톱 브라우저 기준으로 우선 구현한다.
- 기술 스택은 Next.js(App Router), TypeScript, Tailwind CSS, Supabase(PostgreSQL), Vercel로 고정된다.
- Supabase MCP가 이미 설정 및 인증된 상태에서 데이터베이스 테이블 생성 작업을 진행한다.
